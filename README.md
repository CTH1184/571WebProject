## What this project is

This repository contains a front-end website for **Women in Electrical & Computer Engineering (WECE)** at the University of Wisconsin–Madison.

The site was generated from a Figma prototype and then refined into a React web app. The goal of the project is to provide a polished, responsive replacement for the organization’s existing WiscWeb site.

This README is written specifically for TAs/graders so you can quickly understand:

- what part of the repo is the actual website,
- how to run it,
- how the site is structured,
- which pieces are fully functional vs. front-end only,
- and which parts of the exported Replit workspace are scaffold rather than core project code.

## High-level architecture

### Main app
The site is a **React + Vite** front end located in:

```text
artifacts/wece-website
```

### Routing
Routing is handled client-side with **Wouter**.

The routes are defined in `src/App.jsx`:

- `/` → Home
- `/about` → About
- `/board` → Board
- `/events` → Events
- `/sponsors` → Sponsors / Industry
- `/contact` → Contact
- `/get-involved` → Get Involved
- fallback → Not Found page

### Styling
The site uses:

- **Tailwind CSS v4**
- custom styling in `src/index.css`
- some Replit/shadcn-generated UI component files in `src/components/ui`

### State / data
This project is currently **front-end only**.

There is **no persistent backend** connected to the website pages at this time. Most content is stored directly in arrays/constants inside the page files.

Examples:
- Board member data is hardcoded in `src/pages/Board.jsx`
- Sponsor/company data is hardcoded in `src/pages/Sponsors.jsx`
- Event cards are hardcoded in `src/pages/Events.jsx`
- FAQ content is hardcoded inside page components
- Contact/Get Involved forms currently show a success state in the UI only; they do not submit to a database or email service

#### NOTE: You may need to run the following command before hand 

### Option 1: Run only the website app

From inside `artifacts/wece-website`:

```bash
pnpm install
PORT=5173 BASE_PATH=/ pnpm dev
```
Then open the local Vite URL.

### Option 2: Run from the workspace root

From the repo root:

```bash
pnpm install
pnpm --filter @workspace/wece-website dev
```

Because of the current Vite config, `PORT` and `BASE_PATH` are expected to be set.

Example:

```bash
PORT=5173 BASE_PATH=/ pnpm --filter @workspace/wece-website dev
```

### Build

From the website folder:

```bash
PORT=4173 BASE_PATH=/ pnpm build
```

or from the workspace root:

```bash
PORT=4173 BASE_PATH=/ pnpm --filter @workspace/wece-website build
```

### Preview production build

```bash
PORT=4173 BASE_PATH=/ pnpm --filter @workspace/wece-website serve
```

## Important deployment note

The current `vite.config.ts` is still Replit-oriented.

It **throws an error unless** both environment variables are defined:

- `PORT`
- `BASE_PATH`

This is fine inside Replit, but it is inconvenient for GitHub/Vercel/Netlify deployment unless those defaults are added.

A cleaner deployment version would be:

```ts
const port = Number(process.env.PORT ?? 5173);
const basePath = process.env.BASE_PATH ?? "/";
```

This is not a logic bug in the page components themselves, but it is a current setup quirk graders should know about.

## Data strategy

At the moment, the site is intentionally simple and does **not** pull from an API or database.

Instead, content is embedded directly into each page component. This made it easier to:

- match the Figma prototype quickly,
- keep the code easy to understand in a class setting,
- and avoid building a backend before the front-end was complete.

### Current tradeoff
This is straightforward for grading/demo purposes, but longer-term maintenance would be better with one of these improvements:

- central `data/` files for board/events/sponsors,
- a CMS/admin panel,
- Google Sheets/Airtable-backed content,
- or a lightweight backend/API.

## Known limitations / things graders should be aware of

1. **Forms are UI-only**  
   Both the Contact and Get Involved forms simulate submission by showing a success state; they do not send/store data.

2. **A lot of content is hardcoded**  
   This includes board members, events, FAQs, and sponsors.

3. **Replit workspace includes extra scaffold**  
   The repo contains generated files and packages beyond what the WECE front end strictly needs.

4. **Vite config still expects Replit-style env vars**  
   `PORT` and `BASE_PATH` are currently required.

5. **Some UI component files are unused**  
   Replit/shadcn generated a large `components/ui` library, but the actual WECE pages only use a subset of that code directly.

6. **Sponsor logos are not implemented as image assets**  
   Sponsor cards currently use icon-based visual placeholders plus real company names/links.

The strongest areas of the project are:

- clean, branded front-end presentation
- responsive multi-page React structure
- clear separation between shared layout and page content

- persistence/backend integration
